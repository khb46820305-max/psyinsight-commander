# 프로젝트 헌법 (CLAUDE.md)

## 1. 프로젝트 개요
- 이 프로젝트는 **AI 조직화 기반 바이브 코딩 시스템**입니다.
- 목표: 사용자(대표)의 자연어 명령을 받아, AI 에이전트 팀을 조직화하고 지휘하여 개발 작업을 수행합니다.
- 핵심 철학: 개발자는 코드를 직접 작성하는 것이 아니라, AI 팀을 '조직화'하고 '지휘'하는 역할을 합니다.

## 2. AI 조직 구조 (Team Orchestration)

### 2.1 조직 구성원 및 역할

#### PM/기획자 (The Planner)
- **담당 모델**: ChatGPT (GPT-4o), Claude 3.7 Sonnet (Web), Gemini 2.0
- **역할**: 
  - 추상적인 아이디어를 구체적인 기획안(PRD)으로 변환
  - 전체 프로젝트의 방향성 설정
  - 메타 프롬프트를 통한 기획 완성도 향상
- **활용 시점**: 프로젝트 초기 기획, 기능 추가 시 요구사항 분석

#### 수석 개발자/아키텍트 (The Architect)
- **담당 도구**: Claude Code (터미널 기반 에이전트)
- **역할**:
  - 복잡한 코드 베이스 이해 및 전체 구조 설계
  - 하위 에이전트에게 작업 지시 또는 직접 코드 수정
  - 프로젝트의 '맥락(Context)' 관리
- **특징**: 프로젝트의 맥락을 가장 잘 이해하며, 실제 파일 생성/수정 담당

#### 실무 개발자/구현 담당 (The Builder)
- **담당 도구**: Replit, Lovable, Cursor, VS Code
- **역할**:
  - 특정 기능 단위(모듈) 빠른 구현
  - UI/UX 제작 및 프로토타이핑
- **특징**: 
  - Replit: 백엔드/DB 연동 강점
  - Lovable: 시각적 UI 프로토타이핑 강점

#### 검증 및 테스터 (The QA)
- **담당 도구**: MS Clarity, Google Analytics, 별도 AI 세션
- **역할**:
  - 구현 결과물이 기획 의도와 일치하는지 검증
  - 버그 발견 및 사용자 행동 분석
- **팁**: 코딩한 에이전트와 별도 세션으로 제3자 리뷰 수행

## 3. 기술 스택 (Tech Stack)
- **Frontend**: React, Vite, Tailwind CSS (웹 앱의 경우)
- **Backend**: Supabase (Auth, Database), Node.js
- **Deployment**: Replit, Cloudflare Pages, GitHub Actions
- **개발 도구**: Cursor, VS Code, Git

## 4. 코딩 원칙 (절대 준수)

### 4.1 필수 사항
- **TDD 지향**: 모든 기능 구현 전, 테스트 코드를 먼저 작성하거나 계획할 것
- **함수형 컴포넌트**: 클래스형 컴포넌트 대신 React Hooks 사용
- **주석 필수**: 복잡한 로직에는 반드시 한글 주석을 달 것
- **에러 처리**: 모든 API 호출에는 try-catch 블록을 포함할 것
- **문서화**: 모든 주요 변경사항은 todo.md에 반영할 것

### 4.2 작업 관리 원칙
- **이터레이션(Iteration) 단위 작업**: 전체를 한 번에 구현하지 말고 3~4개의 이터레이션으로 분할
- **자동 업데이트**: 작업 완료 시 반드시 todo.md를 업데이트할 것
- **상태 추적**: 완료된 작업은 [x], 진행 중은 [ ]로 표시

## 5. 금지 사항 (Don'ts)
- 사용자의 명시적 허락 없이 기존 파일을 삭제하지 말 것
- 불필요한 라이브러리를 임의로 설치하지 말 것
- 보안상 중요한 정보(API 키 등)를 프론트엔드 코드에 노출하지 말 것
- todo.md나 CLAUDE.md를 임의로 삭제하거나 구조를 크게 변경하지 말 것

## 6. 자주 쓰는 명령어
- 개발 서버 실행: `npm run dev`
- 테스트 실행: `npm test`
- 빌드: `npm run build`
- Git 상태 확인: `git status`
- Git 커밋: `git commit -m "메시지"`

## 7. 작업 흐름 (Workflow)
1. **명령 수신**: 사용자(대표)로부터 자연어 명령 수신
2. **역할 분배**: 명령의 성격에 따라 적절한 AI 역할(Planner/Architect/Builder/QA) 선택
3. **작업 수행**: 선택된 역할에 따라 작업 실행
4. **문서 업데이트**: todo.md에 진행 상황 반영
5. **검증**: QA 역할로 결과물 검증
6. **피드백**: 사용자에게 결과 보고 및 다음 단계 제안

## 8. 보안 가이드라인
- API 키는 환경 변수(.env)로 관리하고 .gitignore에 포함
- 민감한 정보는 절대 하드코딩하지 말 것
- 배포 전 보안 리뷰 요청: "이 코드가 보안상 안전한지 리뷰해줘"

## 9. 에러 대응 전략
- AI가 고치지 못하는 에러 발생 시: "스스로 문제의 원인을 분석하고, 수정 계획을 세운 뒤 실행해(Thinking Token 활용)"
- Git을 활용하여 코드가 꼬였을 때 이전 상태로 롤백 가능하도록 설정
- 에러 로그를 AI에게 제공하여 정확한 진단 요청
