# AI 조직 구조 문서 (ORGANIZATION.md)

## 개요
이 문서는 바이브 코딩 시스템에서 사용되는 AI 에이전트 팀의 조직 구조와 각 역할의 상세한 책임을 정의합니다.

---

## 조직도

```
                    [사용자: 대표]
                         |
                         | (자연어 명령)
                         |
            [명령 라우터 / 역할 분배기]
                         |
        +----------------+----------------+----------------+
        |                |                |                |
    [Planner]      [Architect]        [Builder]        [QA]
   (기획자)      (수석 개발자)      (실무 개발자)    (검증자)
```

---

## 역할별 상세 정의

### 1. PM/기획자 (The Planner)

#### 담당 모델
- ChatGPT (GPT-4o)
- Claude 3.7 Sonnet (Web 버전)
- Gemini 2.0

#### 주요 책임
1. **요구사항 분석**
   - 사용자의 추상적인 아이디어를 구체적인 요구사항으로 변환
   - 기능 명세서 작성
   - 사용자 스토리(User Story) 작성

2. **PRD 작성**
   - 제품 요구 사항 문서(Product Requirements Document) 생성
   - 기능 우선순위 결정
   - 기술적 제약사항 파악

3. **프로젝트 방향성 설정**
   - 전체 프로젝트의 목표와 범위 정의
   - 단계별 마일스톤 설정
   - 리스크 분석

#### 활용 시점
- 프로젝트 초기 기획 단계
- 새로운 기능 추가 요청 시
- 프로젝트 방향 전환 시

#### 프롬프트 예시
```
"이 아이디어를 실현하기 위한 PRD를 작성해줘. 
기능 명세, 사용자 스토리, 테스트 케이스를 포함해줘."

"전문가 관점에서 이 기획의 허점을 비판해줘."

"이 문제를 해결하기 위한 단계적인 사고 과정을 알려줘."
```

### 2. 수석 개발자/아키텍트 (The Architect)

#### 담당 도구
- **Claude Code** (터미널 기반 에이전트) - 주력
- Cursor (코드 편집 및 디버깅)

#### 주요 책임
1. **시스템 아키텍처 설계**
   - 전체 코드베이스 구조 이해
   - 모듈 간 의존성 관리
   - 확장 가능한 구조 설계

2. **코드 구조 관리**
   - 파일 및 폴더 구조 설계
   - 코딩 컨벤션 준수 감시
   - 리팩토링 계획 수립

3. **작업 지시 및 통합**
   - Builder에게 구체적인 작업 지시
   - 여러 모듈의 통합 관리
   - 코드 리뷰 및 품질 관리

4. **문서 관리**
   - CLAUDE.md 준수 확인
   - todo.md 업데이트
   - 기술 문서 작성

#### 특징
- 프로젝트의 '맥락(Context)'을 가장 잘 이해
- 실제 파일 생성/수정 권한 보유
- 다른 에이전트들의 작업을 통합하고 조율

#### 활용 시점
- 복잡한 코드베이스 수정이 필요할 때
- 여러 파일에 걸친 변경이 필요할 때
- 프로젝트 구조 재설계가 필요할 때

---

### 3. 실무 개발자/구현 담당 (The Builder)

#### 담당 도구
- **Replit**: 백엔드/DB 연동, 모바일 앱 개발
- **Lovable**: UI/UX 프로토타이핑
- **Bolt**: 초기 UI 디자인
- **Cursor/VS Code**: 코드 편집

#### 주요 책임
1. **기능 구현**
   - Architect의 지시에 따라 특정 기능 구현
   - UI 컴포넌트 제작
   - API 엔드포인트 개발

2. **프로토타이핑**
   - 빠른 프로토타입 제작
   - 시각적 피드백 제공
   - 사용자 테스트용 MVP 제작

3. **외부 서비스 연동**
   - Supabase, Firebase 등 DB 연동
   - API 통합
   - Replit Connectors 활용 (구글 시트, 독스 등)

#### 특징
- 빠른 구현 속도
- 시각적 결과물 제공
- 특정 도메인(UI/백엔드)에 특화

#### 활용 시점
- 특정 기능의 빠른 구현이 필요할 때
- UI/UX 디자인이 필요할 때
- 외부 서비스 연동이 필요할 때

---

### 4. 검증 및 테스터 (The QA)

#### 담당 도구
- **MS Clarity**: 사용자 행동 분석
- **Google Analytics**: 데이터 분석
- **별도 AI 세션**: 제3자 리뷰

#### 주요 책임
1. **기능 검증**
   - 구현된 기능이 기획 의도와 일치하는지 확인
   - 사용자 스토리 기반 테스트
   - 엣지 케이스 테스트

2. **버그 발견**
   - 코드 리뷰를 통한 잠재적 버그 발견
   - 에러 로그 분석
   - 보안 취약점 검사

3. **사용자 경험 분석**
   - 사용자 행동 데이터 분석
   - 성능 모니터링
   - 개선 사항 제안

#### 특징
- 객관적이고 비판적인 시각
- Builder와 별도 세션으로 독립적 검증
- 데이터 기반 의사결정

#### 활용 시점
- 기능 구현 완료 후
- 배포 전 최종 검증
- 사용자 피드백 분석 시

## 작업 흐름 (Workflow)

### 일반적인 작업 흐름

```
1. [사용자] 자연어 명령 입력
   ↓
2. [시스템] 명령 분석 및 역할 선택
   ↓
3. [Planner] (필요시) PRD 작성 및 기획
   ↓
4. [Architect] 구조 설계 및 todo.md 업데이트
   ↓
5. [Builder] 실제 구현
   ↓
6. [QA] 검증 및 리뷰
   ↓
7. [Architect] 통합 및 최종 확인
   ↓
8. [사용자] 결과 확인 및 피드백
```

### 간단한 작업의 경우
- Planner 단계 생략 가능
- Architect가 직접 Builder 역할 수행 가능
- QA는 선택적

## 역할 선택 가이드

### Planner를 선택하는 경우
- ✅ 새로운 프로젝트 시작
- ✅ 큰 기능 추가
- ✅ 방향성 재검토 필요
- ✅ 요구사항이 모호함

### Architect를 선택하는 경우
- ✅ 복잡한 코드베이스 수정
- ✅ 여러 파일 동시 수정
- ✅ 프로젝트 구조 변경
- ✅ 리팩토링 필요

### Builder를 선택하는 경우
- ✅ 특정 기능 빠른 구현
- ✅ UI 컴포넌트 제작
- ✅ 외부 서비스 연동
- ✅ 프로토타입 제작

### QA를 선택하는 경우
- ✅ 구현 완료 후 검증
- ✅ 버그 수정 요청
- ✅ 코드 리뷰 필요
- ✅ 성능 최적화 검토

---

## 협업 원칙

1. **명확한 책임 분리**: 각 역할은 자신의 전문 영역에 집중
2. **문서 기반 소통**: todo.md와 CLAUDE.md를 통한 소통
3. **단계적 검증**: 각 단계마다 QA를 통한 검증
4. **사용자 피드백 우선**: 사용자의 피드백을 최우선으로 반영

---

**이 조직 구조는 유연하게 조정 가능하며, 프로젝트의 성격에 따라 역할을 추가하거나 변경할 수 있습니다.**
